# 设计文档（≤2 页）

> 作品：知道"该替你决定多少"、且服务整个"局"的本地活动规划执行 Agent
> 一句话：同一道题，我们站高一层——不答"怎么把行程排好、执行掉"，而答"**一个 Agent 到底该替你做多少决定**"。

## 0. 核心洞察

赛题埋着一个内在矛盾：一边要"一键帮你做完"（别让我动手），一边用"递给老婆/发朋友看看"暗示"别自作主张"（我要审）。
全自动违背第二条，事事都问违背第一条。**唯一同时满足两条的，是按"置信度 × 代价"决定自动到什么程度。** 这就是本作品的大脑。
"递给别人看"还藏着第二个信号：**决策不是一个人做的**——于是升维为**群体决策协调器**，而它与主轴是同一个引擎。

## 1. Planning 策略

一句话目标经过一条轻量决策循环（手搓，不用 LangChain，避免标准 ReAct 框死自定义置信度逻辑）：

```
目标 → [意图解析] 列出"已知/未知"清单（不默默填默认值）
     → [置信度×代价] 对每个待定决定评估，推导 做/建议/问
     → [内部冲突协调] 在"无唯一正确答案"处带理由权衡（娃 vs 老婆）
     → 第一版 Plan（每个决定带 confidence/basis/cost/reasoning/disposition）
     → A 回答 ask 类问题 + 对 suggest 类拍板
     → [协同评审回合] 多人各自改 → 自动合并无争议 → 撞车处给带理由建议 → A 裁决 → v2
     → A 最终确认 → 执行 + 异常兜底 → send_plan
```

**决策大脑（`rules.py`）**。`disposition = f(confidence, cost)`：

| 条件 | 处置 |
|---|---|
| `cost==high` 或 `confidence<0.4` | **ask**（停下来问） |
| `confidence≥0.7` 且 `cost==low` | **auto**（直接做） |
| 其余 | **suggest**（给建议） |

置信度可解释：餐厅按"命中了几条已知约束（儿童餐/低卡/距离/过敏/预算）"加减得出，并把**缺失项**（如预算未知）显式压低置信度——这正是"先承认不确定"的认知起点。
评审阶段，"采纳某条改动"的置信度 = `0.55×内容契合度 + 0.45×作者 weight`：**谁改的**也是输入，老婆改的餐厅天然比不熟的朋友更可信。
置信度**会更新**：低置信决定被某人的反馈确认/推翻后重算，并在 `confidence_basis` 记下"被谁修正"。

**协同评审（`review.py`，升维亮点）**。单机用"视角切换"模拟多人（`as 老婆`），改动是真改 Plan、`author` 真记录——只有"分享到另一台设备"用切视角代替，**绝不搭实时同步/WebSocket**（那是工程奇观，无评分价值、最易现场翻车）。
无争议改动**自动吸收**（如"海鲜过敏"→ 纳入约束并剔除海鲜店）；多人改到同一决定**拎成 Conflict**，给"采纳谁 + 为什么（含 weight）"的建议，拍板权留给 A。

**情感辅线**是主轴的一个特例，不是第二条线：发现"老婆减肥"→ 推断送高热量蛋糕适得其反 → 把礼物建议成"花 + 展览票"，但 `disposition=suggest`——情绪敏感的事置信度到不了"直接做"，所以走"建议"。**一个引擎，主辅一体。**

**AI 层（可选增强，`llm.py` / `llm_tools.py`）**：意图解析与候选生成可切到真实 LLM（OpenAI 兼容、标准库直连、零依赖），把任意自然语言听成结构化约束、按目标实时生成逼真候选（带齐决策字段）。**关键纪律：AI 只负责"听懂 + 找店"，`置信度×代价→做/建议/问` 仍由确定性大脑算**——认知主轴不进黑盒、可解释、可复现。调不通自动回退本地 Mock 库，且带磁盘缓存（命中秒回、可离线复演），**现场不把命运交给网络**。

## 2. 工具调用链路（`tools.py`，全 Mock）

覆盖"查—订—买—送—发"五类（贴合美团能力），返回数据**带齐决策所需字段**（人均/儿童友好/低卡/过敏原/距离/排队/等位时长），大脑靠这些算置信度：

- **查**：`search_restaurants`（带过滤）/`search_activities`/`check_availability`/`get_route`
- **订**：`book_restaurant`/`join_queue`　**买**：`order_cake`/`order_flowers`/`buy_ticket`　**送**：`arrange_delivery`　**发**：`send_plan`

依赖顺序执行：**订餐 → 下礼物 → 送花 → 发计划**。每笔成功下单累加 **GMV 计数器**（屏幕实时可见）。最后 `send_plan` 产出一张**移动端可分享方案卡**（`card.py`），即赛题"把手机递给老婆 / 发给小张看"的那一屏——家人愿意打开、可一键复制分享文案。
Mock 用固定随机种子 + `forced` 强制项，保证演示**可复现**，并能主动制造"信息缺失/满了/超时/失败"。

**增量 GMV 从哪来（商业价值硬论证，不只是数钱）**。GMV 计数器对应三条真实增量：
1. **激活低谷时段**——家庭"周六下午 3–6 点"本是本地消费转化低谷（缺好入口）；Agent 把"帮我安排"变成一个**确定性消费入口**。
2. **发现型订单**——满座兜底 / 口味协调时，Agent 推荐用户"本来不会发现"的店，这笔是**纯增量**。
3. **组局乘数**——一句话撬动门票 + 餐厅 + 礼物 + 用车多个 SKU；组局（≥3 人）客单天然高于个人。

> 一次成功执行平均触发约 3 个 SKU 下单，把"懒得安排"的沉默、零散、偶发需求，转成可被美团履约网络承接的**结构性流量入口**。护城河 = 数据规模 × 履约网络，只有美团生态能同时给齐——这是别的平台抄不走的长期价值。

## 3. 异常处理（多数队伍的死穴 = 我们的得分点）

执行是**带状态的流程**（待执行/执行中/成功/失败），三类异常全做且能演：

1. **餐厅满了** → 自动 `join_queue` 报等位 ETA；ETA 超过阈值（30min）→ 从满足约束的候选里**自动改订备选**。
2. **工具超时** → 仅对 `timeout` 类**自动重试**（演示：下花单超时一次后重试成功）。
3. **中途某步失败 → 事务完整性**：把"送花"视为一笔事务 `[下花单 → 送花到店]`，第二步失败时**回滚第一步已下的花单**（退款，GMV 相应回退），并给出**补偿**（建议到店现买，不影响惊喜）。独立的展览票事务不受影响。
4. 意图模糊 → 反问澄清（ask 类决定即此机制）；下单/支付失败 → 回滚。

> "顺风顺水能跑"是门槛，"出事了还能优雅稳住"才出彩。

## 4. 取分对应

| 维度 | 靠什么拿分 |
|---|---|
| 创新性 | ①"按置信度决定自动到什么程度"重定义这道题；② 升维为群体决策协调器；③ 补全用户没说的偏好 |
| 完整性 | 闭环全（查订买送发）+ 协同评审回合 + 异常兜底/回滚 |
| 应用效果 | 内部冲突协调 + 多人分歧协调 + 该问才问 + 情感体贴 |
| 商业价值 | 一句话撬动多笔交易；群体场景 = 更高客单；激活"懒得安排"的沉默需求；美团履约网络做护城河 |

## 5. 工程纪律（始终遵守）

- **守灵魂字段**：`confidence/confidence_basis/cost/reasoning/disposition` 任何时候不为"先跑通"而删（`selftest.py` 自动守护，83 项断言）。
- **协同评审走轻量**：视角切换 + `author` 字段，绝不做真多端/实时同步。
- **置信度规则先粗后细**：第一版规则函数能跑就行，不陷入调参完美主义。
- **每加一功能先问**：能否体现"按置信度决定 做/建议/问"或"服务整个局"？不能则砍掉或降为配角。
